# Ch.5 — 코드는 논문의 주장과 맞물릴 때 읽힌다

SLAM과 CV 분야에서 논문만 읽는 일은 절반만 읽는 일에 가깝다. 좋은 논문은 대개 code repository, project page, dataset, issue thread를 함께 남긴다. 하지만 코드를 열었다고 해서 더 깊이 읽은 것은 아니다. 코드를 읽는 목적은 repository 구조를 외우는 일이 아니라, 논문의 claim이 실제 구현의 어느 결정과 맞물리는지 확인하는 일이다.

한 편을 코드와 함께 읽는 사이클은 느리다. 먼저 paper를 1st, 2nd pass로 본다. 이 단계에서 message가 잡히지 않으면 repository를 열어도 얻는 것이 적다. 다음으로 README와 entry script를 훑어 어디서 시작해 어디로 흐르는지 한 줄로 적는다. 그 다음 직접 빌드하고 한 번 돌려 본다. build error는 예외가 아니라 작업의 일부다. 마지막으로 paper의 방법 단락과 코드의 함수가 어떻게 대응되는지 찾는다.

이 대응은 모든 줄을 맞추는 작업이 아니다. 결정적인 한두 줄이면 충분하다. 논문에서 “we use a robust kernel”이라고 쓴 문장이 실제로 어느 loss function에 들어가는지, “pre-integration”이라고 부른 과정이 어느 module에서 수행되는지, figure의 pipeline 박스가 코드의 어느 directory로 갈라지는지 찾는다. 그 지점이 잡히면 논문은 설명이 아니라 작동하는 물건이 된다.

issue와 PR도 읽기 자료다. paper는 성공한 경로를 정리하지만 issue는 실패한 경로를 남긴다. build가 자주 깨지는 곳, hyperparameter가 바뀐 이유, multi-threading을 포기한 이유가 issue와 PR에 남아 있다. 논문의 limitation 섹션은 저자가 인식한 한계이고, open issue는 사용자가 부딪힌 한계다. 둘을 함께 보면 방법의 실제 범위가 보인다.

모델은 paper-code 매핑에서 강력하다. “이 문장이 코드의 어느 함수에 해당하는가”를 물으면 후보를 빨리 좁혀 준다. 하지만 line 번호는 쉽게 틀린다. 비슷해 보이는 함수를 짚고 실제로는 다른 모듈일 때도 있다. 후보를 사람이 검증하지 않으면, 6개월 뒤 잘못된 line 번호 위에 자기 실험을 세우게 된다.

수식도 같은 방식으로 읽는다. proof나 derivation을 모두 외우는 것이 아니라, 어떤 가정이 어느 식으로 들어가고 어느 코드 결정으로 내려오는지 본다. 수식이 논문의 주장과 코드의 실행 사이에서 다리 역할을 할 때, 읽기는 비로소 깊어진다.

좋은 연구자는 코드를 직접 다 쓰는 사람이 아니다. 또 모든 코드를 자동화 도구에 맡기는 사람도 아니다. 논문의 주장, 수식의 가정, 코드의 결정이 어디에서 만나는지 알아보는 사람이다. 10년 후에도 남는 구현 감각은 특정 library 사용법이 아니라 이 매핑 능력에서 온다.
